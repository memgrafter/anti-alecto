### TL;DR
PiSwarm is a GitHub automation framework that uses the `pi` agent plus git worktrees to run issues, PR reviews, epics, and whole project plans in parallel with state persistence, retries, and crash recovery.

### Key Quote
“Parallel GitHub issue and PR processing using the `pi` agent and Git worktrees.”

### Summary
- **What it is**
  - A tool for orchestrating parallel work on GitHub issues and pull requests using the `pi` agent.
  - Designed around isolated **git worktrees** so each agent/task works independently.
  - Supports multi-level orchestration:
    - **swarm.sh** for issue waves
    - **pr-swarm.sh** for PR review/fixes
    - **captain.sh** for epic-level orchestration
    - **commander.sh** for milestone/project-level orchestration

- **Core features**
  - Parallel processing of multiple issues and PRs
  - Dependency-aware “wave” execution for epics
  - Project generation mode that can create GitHub issues from a spec
  - Headless structured execution via `pi --mode json`
  - Resume support with `--resume`
  - Automatic retries with exponential backoff
  - Fatal error detection for quota/auth/rate limit cases
  - Lock files and heartbeat monitoring to prevent duplicate runs and detect crashes
  - Optional auto-creation of issues in project mode

- **Command hierarchy**
  - **Commander**: top-level orchestrator for projects or multiple epics
  - **Captain**: orchestrates one epic by parsing sub-issues, grouping them into parallel-safe waves, running swarms, reviewing PRs, and validating success criteria
  - **swarm.sh**: executes issue tasks in parallel, commits work, creates PRs
  - **pr-swarm.sh**: reviews/fixes PRs in parallel and posts comments

- **Installation prerequisites**
  - `gh` CLI, authenticated
  - `jq`
  - `pi` agent
  - `git`
  - Clone the repo and optionally symlink it into `~/.config/agents/skills/pi-swarm`

- **Typical usage**
  - Work on issues:
    - `scripts/swarm.sh 12 15 22`
  - Review PRs:
    - `scripts/pr-swarm.sh 101 105`
  - Run an epic:
    - `scripts/captain.sh --epic 151`
  - Run multiple epics:
    - `scripts/commander.sh --epics 151 160 175`
  - Generate and execute a new project:
    - `scripts/commander.sh --project "Build a CLI todo app with SQLite"`

- **Important options**
  - Common flags include `--model`, `-j/--jobs`, `--dry-run`, `--resume`, `--force`
  - Script-specific options control pushing, cleanup, timeouts, wave limits, parallelism, merging PRs, and project/milestone selection

- **Monitoring and state**
  - Logs and outputs are stored under:
    - `.worktrees/`
    - `.captain/`
    - `.commander/`
  - State files track:
    - task status (`pending`, `in_progress`, `completed`, `failed`, `fatal`)
    - retries
    - errors
    - wave completion
  - Heartbeat files are used to detect stale processes every 30 seconds

- **Error handling**
  - Retries:
    - rate limits
    - timeouts
    - network issues
    - 5xx API errors
  - Stops immediately on fatal conditions:
    - auth failures
    - quota/billing limits
  - Errors are recorded in state files and surfaced in summaries

- **Issue/epic/project formats**
  - Supports epics written as:
    - linked sub-issues
    - task lists that get converted into GitHub issues
    - simple descriptions treated as a single task
  - Recommends adding:
    - sub-issues
    - parallelization strategy
    - success criteria

- **Repository structure / output**
  - Provides examples of generated logs, state files, and PR URL files in `.worktrees/`, `.captain/`, and `.commander/`
  - Includes architecture diagrams showing Commander → Captains → swarm/pr-swarm execution layers
