Phase 1 Summary
The Core Problem: Agent ≠ LLM Call
Three of seven Minsky patterns (K-lines, pronomes, guard learning) were postponed because we framed the problem as a granularity gap — "we need more atomic agents than our heavyweight LLM calls."
But this framing was wrong. We already have atomic agents. We've been building them throughout Phase 1 without naming them:
- Hook actions — the censor in Example 1 (regex pattern matching, zero LLM calls)
- Rule-based evaluators — the B-brain's loop/stall/cost detectors in Example 4
- Individual tool calls — each tool invocation in a
tool_loop
is individually checkpointed viaon_tool_calls
/on_tool_result
- Heuristic observers — B-brain checkpoint polling, state history tracking
The real gap isn't "we need more atomic agents" — it's that we haven't connected our existing non-LLM agents to the Minsky observation/capture/signaling infrastructure. Tool-call sequences are already individually checkpointed. Hook actions already fire on every state transition. We just haven't wired them into K-line capture or pronome signaling.

The Three Paths
Path A: Phase 2 As-Designed
Build KLineStore, guard learning, uniframe generator on top of context-dict snapshots. Accept that K-lines are thin.
Low risk Low reward
Path B: Recognize What Exists
Formalize the agent taxonomy. Wire existing non-LLM agents (tools, hooks, rules) into K-line capture and uniframe slots. No new infrastructure needed.
Low risk High reward
Path C: New Core Primitives
Agent pools, typed signal classes, dynamic specialist selection. High effort, high reward, but only justified by product need.
High risk High reward

Recommendation
Immediate: Formalize agent taxonomy + wire to K-lines
This is naming and a hooks class, not new infrastructure:
- Agent taxonomy — document that FlatMachines has five agent types (LLM, hook action, tool call, rule-based evaluator, observer). These are already built. Name them.
- K-line hooks from tool sequences — a
KLineHooks
class that reads existingtool_loop_state
from checkpoints and writes to a K-line table. Records which tools were called, in what order, for what task. This is reading existing data, not building new capture infrastructure. - Revisit Example 3 — K-lines become meaningful when they capture tool activation patterns + hook action firings, not just context dict snapshots.
Effort: Low (hooks class + naming) Unlocks: Examples 3, 7 + Phase 2 KLineStore + guard learning
Short-term: Uniframe generator with typed slots
The uniframe generator becomes genuinely interesting when combined with the agent taxonomy:
- Extract coordination skeletons from similar machine configs
- Classify each state's agent type (LLM / hook / tool / rule)
- Identify which slots accept multiple agent types (tolerance boundaries)
- Surface the question: "is this slot fundamentally LLM-reasoning, or could a cheaper agent type fill it?"
Effort: Low (offline script) Unlocks: cost optimization + frame theory
Medium-term: Typed signal classes (Phase 3, item #11)
Add signal_class
to the Signal dataclass. This is the first actual core spec change needed:
- Polyneme vs isonome distinction as first-class runtime behavior
- Microneme ambient context (cumulative signals)
- Foundation for pronome tracking (once parallel agencies exist)
Effort: Medium (core spec) Unlocks: pronome tracking prerequisites
Long-term: Agent pools + parallel agencies
These remain the high-effort items. They require core spec work (dynamic agent selection, launch N of M) and convention work (template families that share state structure).
Defer until there's a concrete product need — not just Minsky fidelity — that requires selecting from a pool of 20+ specialists.
Effort: High (core spec) Unlocks: full Minsky pattern library
