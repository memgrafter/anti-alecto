🎯 Priority Gap Analysis
What Exists
user_actions.priority
— integer column with index, but every row is 1 or 2 (binary, not a real queue)- Git history shows priority discussed in 3 early commits (Feb 21-22): gap analysis, lifecycle criteria, model selection

) serve as coarse human triage
What's Missing
- No plan-level priority:
docs
table has no priority/urgency column — plans are implicitly ordered by which review the human responds to first - No task-level priority: tasks have status but no ordering signal for the swarm to pick the most important work
- No execution priority: when multiple executions are waiting for signals, nothing determines which resumes first
- No automated triage: stalled executions and aged-out drafts accumulate silently — no alerts, no auto-cleanup
Impact
Priority is currently an emergent human behavior, not a system property. The operator's attention is the scheduler.
This works at low plan counts but creates invisible debt — 6 stalled executions, 2 reviews aging 14+ days,
7 draft plans with no path forward. A priority field on docs
+ automated staleness alerts would
convert this implicit knowledge into actionable system state.
📋 Key Insights
✅ Working Well
- Changeset system: 96% commit rate (330/344), 4% abandon rate is healthy
- Dual-checker pattern: parallel GPT+ZAI review catches more issues
- Signal-driven resume: checkpoint-based persistence allows long-lived workflows
- Task decomposition: plan 104 produced 29 tasks, executed 5, showing the pipeline works end-to-end
- Swarm execution: plan 134 completed all 5 tasks across 5 workers in 4 minutes
⚠️ Needs Attention
- Execution gap: 9 of 11 active plans have zero task runs — pipeline produces plans faster than execution capacity
- Stalled executions: 6 non-terminated executions from Mar 11-18 need manual cleanup
- Review bottleneck: bimodal response (instant or hours) blocks entire pipeline
- Draft rot: 7 draft plans aging 18-30 days with no path forward
- No priority system: human attention is the only scheduler — doesn't scale
