Your example is on the extreme end of what I've witnessed, but this reminds me of so many professional discussions. A key property of a (good) function contract is that it should be unambiguous who's to blame (caller or callee) when something goes wrong, and when you don't specify function contracts, type invariants, etc, people fall back on inevitably divergent and frequently strange headcanon.
I don't think this directly applies in your std::string example since the relevant standard library UB is specified, but it might explain why the person in question was used to thinking this way.
Example: I've worked with a surprising number of C programmers whose apparent headcanon is that if a function takes a const char * parameter for a zero-terminated string then (1) it should be valid to pass NULL and (2) a NULL pointer should behave like an empty string.

Oh that std::string one sticks with me simply because of how vehemently opposed to the idea that "reading outside of the bounds of the string is UB" the reporters were - literally they updated their test case to do a capacity check to ensure "they weren't reading outside of the buffer" and so weren't invoking UB. Basically they wanted a resize to be valid for security sensitive erasure, which it is not, and just refused to accept that reading outside of bounds being UB meant we did not need to allocate a new object. I can't recall the exact details but basically they said
-
The spec says "operation X creates a new string and assigns it to A"
-
Therefore A is replaced
-
Therefore the old data in A is no longer accessible
And just refused to accept that being able to observe that observing that a new allocation did not happen, or the memory was not cleared, required UB, and therefore any code that did so was definitionally wrong. It was an absurdly long back and forth that took more than a fortnight in which they kept changing their test case to "remove UB" while refusing to understand that the act of accessing beyond the string size was UB, no matter what the capacity was.
