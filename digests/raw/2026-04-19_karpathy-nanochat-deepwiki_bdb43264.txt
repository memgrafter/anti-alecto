This page provides a high-level introduction to nanochat: its purpose as a minimal full-stack LLM training system, the single-complexity-dial design philosophy, and the Time-to-GPT-2 leaderboard concept that drives development. For detailed instructions on installation and running your first training, see Getting Started For in-depth documentation of specific subsystems, consult the relevant sections (e.g., Base Model Pretraining Model Architecture Data Pipeline).

nanochat is a minimal experimental harness for training large language models from scratch. It is designed to be:
- Complete: Covers the entire LLM lifecycle including tokenization, pretraining, supervised fine-tuning (SFT), reinforcement learning (RL), evaluation, and deployment README.md1-6
- Accessible: Runs on a single GPU node (typically 8xH100 or 8xA100) README.md32-35
- Minimal: The codebase is small, readable, and hackable with no configuration objects or framework abstractions README.md1-6
- Cost-effective: Trains a GPT-2 capability model in ~1.65 - 1.8 hours for approximately $40-$50 (vs. $43,000 in 2019) README.md21-24
The repository enables you to train your own GPT-2 capability LLM and interact with it through either a command-line interface (scripts/chat_cli.py
) or a ChatGPT-like web interface (scripts/chat_web.py
) README.md54-60
| Stage | Script | Output | Description |
|---|---|---|---|
| Tokenization | scripts/tok_train.py | tokenizer.model | Trains BPE tokenizer with 32,768 vocab on dataset |
| Base Pretraining | scripts/base_train.py | Base checkpoint | Trains transformer from scratch using scaling laws |
| SFT | scripts/chat_sft.py | Chat checkpoint | Adapts base model for conversation using SmolTalk/MMLU/GSM8K |
| RL (Optional) | scripts/chat_rl.py | Aligned checkpoint | Further aligns model using GRPO/SimPO |
| Evaluation | scripts/base_eval.py , scripts/chat_eval.py | Metrics | Measures CORE score, val_bpb, task accuracy |
| Inference | scripts/chat_cli.py , scripts/chat_web.py | Interactive chat | Deploys model for user interaction with streaming API |

nanochat's defining design principle is the single complexity dial: the --depth
argument (number of transformer layers) automatically determines all other hyperparameters to produce compute-optimal models README.md6-8 Users specify only the model size they want; the system calculates:
- Model width (
n_embd
), number of attention heads (n_head
) dev/LEADERBOARD.md29-37 - Learning rates (base LR, embedding LR, with
dmodel_lr_scale
) - Training horizon (total tokens based on scaling laws via
target-param-data-ratio
) dev/LEADERBOARD.md36-37 - Batch size (optimal for the given depth) dev/LEADERBOARD.md32
- Weight decay schedules and Muon momentum schedules
This is implemented through scaling law formulas in scripts/base_train.py
that map depth to optimal hyperparameters. The result is that sweeping --depth
produces a miniseries of compute-optimal models at various scales README.md77-87
| Depth | Approximate Capability | Parameters | Training Time (8xH100) |
|---|---|---|---|
| 12 | GPT-1 scale | ~124M | ~5 minutes |
| 16 | Intermediate | ~220M | ~15 minutes |
| 20 | Approaching GPT-2 | ~343M | ~45 minutes |
| 24-26 | GPT-2 | ~475-600M | ~1.65 - 1.8 hours |
The philosophy eliminates the need for exhaustive hyperparameter tuning: any improvement to the codebase must work across all depths, ensuring principled changes rather than single-model overfitting dev/LEADERBOARD.md51-52

nanochat uses an explicit precision system rather than PyTorch's autocast
dev/LOG.md60-67 The global COMPUTE_DTYPE
variable is auto-detected based on hardware:
| Hardware | Default COMPUTE_DTYPE | Rationale |
|---|---|---|
| CUDA SM 80+ (A100, H100) | bfloat16 | Native BF16 tensor cores dev/LOG.md71 |
| CUDA SM < 80 (V100, T4) | float32 | No BF16; FP16 available via NANOCHAT_DTYPE=float16 dev/LOG.md71 |
| CPU / MPS | float32 | No reduced-precision tensor cores dev/LOG.md71 |
Override with environment variable:
Optional FP8 training is available via --fp8
flag on Hopper+ GPUs, providing speedups using torchao
tensorwise fp8 scaling dev/LEADERBOARD.md116-117

nanochat makes several architectural choices that distinguish it from typical LLM frameworks:
- No autocast: Explicit
COMPUTE_DTYPE
and a customLinear
class that casts weights to match input dtype in forward pass replace "magic" precision management dev/LOG.md60-75 - ClimbMix Dataset: Switched from FineWeb-EDU to NVIDIA ClimbMix 400B, reducing GPT-2 speedrun time by 27% (from 2h46m to ~2h) dev/LOG.md95-98
- Hybrid Optimization: Uses
Muon
for matrix parameters (orthogonalization) andAdamW
for embeddings and scalars to optimize wall-clock capability gains dev/LOG.md17-18 - Flash Attention 3: Support for Hopper-optimized kernels (FA3) with automatic SDPA fallback for older hardware or fp16/fp32 paths dev/LOG.md90-92
- SoftCap Tuning: Default logit softcapping set to 20 based on d24 scale experiments, improving validation loss dev/LOG.md118-121
