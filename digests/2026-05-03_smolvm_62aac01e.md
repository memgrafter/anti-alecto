### TL;DR
Smolvm is a cross-platform CLI for running isolated Linux microVMs locally, packing them into portable `.smolmachine` artifacts, and managing persistent or ephemeral workloads with features like network allowlisting, SSH agent forwarding, and optional GPU acceleration.

### Key Quote
"Ship and run software with isolation by default."

### Summary
- **What it is**
  - `smolvm` is a CLI tool for creating and running custom Linux virtual machines locally.
  - It targets **macOS and Linux** and emphasizes **hardware isolation**, **fast boot**, and **portable packaged machines**.

- **Core capabilities**
  - Run ephemeral commands in a clean VM that is removed after exit.
  - Create persistent machines that can be started, stopped, and reused.
  - Package a machine into a single **`.smolmachine`** file for reuse on supported hosts.
  - Support **OCI images** (Docker-compatible image format) without requiring a Docker daemon.

- **Install and quick start**
  - Installation command:
    - `curl -sSL https://smolmachines.com/install.sh | bash`
  - For discovery of commands:
    - `curl -sSL https://smolmachines.com/install.sh | bash && smolvm --help`
  - Example ephemeral run:
    - `smolvm machine run --net --image alpine -- sh -c "echo 'Hello world from a microVM' && uname -a"`
  - Example interactive shell:
    - `smolvm machine run --net -it --image alpine -- /bin/sh`

- **Main use cases**
  - **Sandbox untrusted code**
    - Network is off by default.
    - Can enable network with `--net` and restrict egress with `--allow-host`.
    - Example: allow only `registry.npmjs.org`, block `google.com`.
  - **Pack into portable executables**
    - Converts workloads into self-contained binaries.
    - Example uses `python:3.12-alpine` and claims boot times under 200 ms.
  - **Persistent development machines**
    - Create a VM, install packages, stop it, and later restart with state preserved.
  - **Git/SSH without exposing keys**
    - Supports forwarding the host SSH agent with `--ssh-agent`.
    - Private keys stay on the host; the guest can use the agent but not extract keys.
  - **Declarative environments**
    - Uses a **Smolfile** in TOML format to define image, networking, allowlists, init commands, volumes, and auth settings.

- **Examples and config**
  - Sample Smolfile:
    - `image = "python:3.12-alpine"`
    - `net = true`
    - `[network] allow_hosts = ["api.stripe.com", "db.example.com"]`
    - `[dev] init = ["pip install -r requirements.txt"]`
    - `[dev] volumes = ["./src:/app"]`
    - `[auth] ssh_agent = true`
  - Example projects linked:
    - Python app
    - Node app
    - Doom web example

- **How it works**
  - Each workload gets its own kernel and runs on:
    - **Hypervisor.framework** on macOS
    - **KVM** on Linux
  - Uses **libkrun** as the VMM with a custom kernel (`libkrunfw`).
  - `.smolmachine` artifacts run anywhere the host architecture matches.
  - Defaults:
    - **4 vCPUs**
    - **8 GiB RAM**
  - Memory is **elastic** via virtio balloon; unused memory is reclaimed by the host.

- **Comparison claims**
  - Compared to containers, Colima, QEMU, Firecracker, and Kata, smolvm positions itself as:
    - per-workload VM isolation
    - fast boot time (`<200ms`)
    - macOS-native support
    - embeddable SDK/library support
    - portable artifacts (`.smolmachine`)
  - The table frames it as more isolated than containers and more portable/embeddable than most VM runtimes.

- **Platform support**
  - macOS Apple Silicon → arm64 Linux guests
  - macOS Intel → x86_64 Linux guests (untested)
  - Linux x86_64 → x86_64 Linux guests via KVM
  - Linux aarch64 → aarch64 Linux guests via KVM

- **Known limitations**
  - Network must be explicitly enabled.
  - Networking supports **TCP/UDP only**, not ICMP.
  - Volume mounts only support **directories**, not individual files.
  - macOS binaries must be signed with Hypervisor.framework entitlements.
  - `--ssh-agent` requires `SSH_AUTH_SOCK`.
  - GPU support requires libkrun built with `GPU=1` plus host GPU/Vulkan support.

- **GPU acceleration**
  - Exposes host GPU to guests using **virtio-gpu / Venus** (Vulkan over virtio).
  - On macOS, required GPU components are bundled.
  - On Linux, users must install `virglrenderer` and Vulkan driver packages.
  - Example CLI:
    - `smolvm machine run --gpu --image alpine -- vulkaninfo --summary`
  - Smolfile example:
    - `gpu = true`
    - `gpu_vram = 2048`
  - Notes a headless Chromium example using ANGLE + Venus for accelerated WebGL.
