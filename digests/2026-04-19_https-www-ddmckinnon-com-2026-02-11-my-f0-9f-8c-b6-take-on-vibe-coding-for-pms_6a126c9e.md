### TL;DR
A Meta PM argues that PMs generally should not spend time landing production diffs at Meta scale because it’s low-leverage and can create tech debt, but coding is still valuable when it helps communicate ideas, understand systems, run realistic experiments, use unique domain knowledge, or simply for fun.

### Key Quote
“PMs landing prod features is generally snacking and mistaking motion for progress (flex on LinkedIn but don’t land any real prod impact).”

### Summary
- The author says they originally shared this internally at Meta in response to “a deluge of clout-chasing posts” praising PMs who ship production code.
- Main thesis: **PMs shouldn’t waste time landing prod diffs at Meta scale** if that work distracts from higher-leverage responsibilities.
- Reasons given for discouraging PMs from doing this:
  - If the feature matters, PMs should **fix the prioritization system** rather than work around it.
  - PMs may code slowly relative to engineers, while still consuming expensive company time.
  - Random PM-driven features can create **technical debt** and even break production, including internal tools.
  - Using senior technical leads for code review is not the best use of their time if the work is not strategically important.
  - Shipping small code changes can become “snacking” — activity that looks productive but doesn’t move the business much.
- The author then gives a more positive answer to **why PMs should code**:
  - To **communicate a feature more clearly**; a prototype often works better than explanation.
  - To **understand systems** better, which improves prioritization and communication with engineers.
  - To run **more realistic experiments** with human volunteers, since static mocks and UX research prototypes can be insufficient.
  - To leverage **unique resources or expertise** a PM may have, such as deep knowledge from a previous company or API domain.
  - Because it can be **fun**.
- Overall tone: opinionated, anti-hype, and focused on leverage and role clarity rather than celebration of coding for its own sake.
