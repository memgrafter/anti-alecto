### TL;DR
This paper is the first large-scale empirical study of agent context files like `CLAUDE.md` and `AGENTS.md`, finding that they are actively maintained, structurally shallow, and heavily focused on build/implementation instructions while largely neglecting security and performance guidance.

### Key Quote
"non-functional requirements like security (14.5%) and performance (14.5%) are rarely specified."

### Summary
- The study analyzes **2,303 agent context files** from **1,925 GitHub repositories** and focuses on Markdown instruction files used by coding agents, such as `CLAUDE.md`, `AGENTS.md`, and `copilot-instructions.md`.
- Main finding: these files behave less like static docs and more like **living configuration artifacts**:
  - **67.4%** are modified in multiple commits
  - changes are usually **small incremental additions**
  - deletions are less common
- Structural pattern:
  - files usually have a **single H1 heading**
  - content is organized into **H2/H3 sections**
  - this shallow hierarchy appears to make them easier for both humans and LLMs to parse
- Content prevalence:
  - **62.3%** include **build/run commands**
  - **69.9%** include **implementation details**
  - **67.7%** include **architecture**
  - only **14.5%** mention **security**
  - only **14.5%** mention **performance**
- Interpretation:
  - developers seem to optimize these files for **getting agents to execute code correctly**
  - they do **not** usually encode broader quality constraints unless a problem forces them to
- Classification experiment:
  - the authors used **GPT-5** to classify content into **16 instruction types**
  - evaluation used **332 manually labeled files**
  - result: **0.79 micro-average F1**
  - strong on concrete functional categories, weaker on abstract ones
    - example: **Architecture = 0.93**
    - **Maintenance = 0.56**
- The paper frames this as a mismatch between what agent context files emphasize and what agentic coding systems often need:
  - strong support for **functionality**
  - weak support for **non-functional requirements** like security, performance, and maintainability
- Stated limitations:
  - classification depends on **GPT-5**, which may limit reproducibility
  - corpus is limited to **Markdown files with specific names**
  - the study is based on **presence/absence** of instruction types, not their depth or quality
- Implications:
  - future work should define and detect **“context debt”**
  - benchmarks should test whether agents actually follow **non-functional requirements**
  - more work is needed on whether richer instructions improve agent output quality
