### TL;DR
This digest summarizes an arXiv paper showing that test-driven code generation can be scaled from functions to classes by using dependency-aware scheduling, method-level public tests, and reflection-based repair, improving class-level correctness across eight LLMs.

### Key Quote
“the authors construct ClassEval-TDD, a cleaned and standardized benchmark with consistent specifications and deterministic tests.”

### Summary
- **Paper title / topic**
  - *Scaling Test-Driven Code Generation from Functions to Classes: An Empirical Study*
  - arXiv ID: **2602.03557**
  - Main claim: class-level code generation can be improved by adapting TDD to handle **intra-class dependencies**.

- **Core method**
  - The framework has **three phases**:
    1. **Dependency analysis** to infer method call relationships and create a **topological generation order**
    2. **Incremental method generation** using **method-level public tests** as executable specifications
    3. **Reflection-style repair** when tests fail, with **up to 3 repair attempts per method**
  - The system prioritizes **callee methods before caller methods** to avoid missing-method or undefined-reference errors.
  - Public tests are used during generation; **private tests** are reserved for final evaluation.

- **Benchmark**
  - Introduces **ClassEval-TDD**, a cleaned and standardized version of ClassEval.
  - It aims to provide:
    - consistent specifications
    - deterministic tests
    - a separation between **public** and **private** tests

- **Key results**
  - Across **eight LLMs**, the framework improves **class-level correctness by 12–26 absolute points**
  - Achieves up to **71% fully correct classes**
  - Average repairs are relatively low: **0.62–1.03 repairs per method**
  - Most failures converge within **1–2 rounds**

- **Why it works**
  - **Dependency-aware scheduling** reduces structural failures by generating methods in an order that satisfies prerequisites.
  - **Method-level tests** create tighter feedback loops and localize errors earlier than whole-class testing.
  - **Reflection-based repair** separates diagnosis from patching, which is especially helpful for smaller models.

- **Evidence and ablations**
  - Removing reflection reduces class success by about **1–4 points**, with bigger impacts on smaller models.
  - The paper also highlights that models often **over-predict dependencies**, which is safer but can add unnecessary ordering constraints.

- **Limitations**
  - **Single-class Python only**: does not cover multi-file systems or other OOP languages like Java/C++.
  - Relies on **method-level public tests**, which may not exist in real-world settings.
  - Benchmark issues include **non-deterministic tests**, **specification mismatches**, and unclear prompts.
  - The paper identifies a persistent **composition gap**: methods may pass individually while the whole class still fails due to shared state or cross-method invariants.

- **Usefulness / relevance**
  - Most valuable if you care about:
    - LLM code generation
    - test-driven prompting / synthesis
    - benchmark design for code models
    - dependency-aware generation strategies
  - Less useful if you need a production-ready system for multi-file repositories, because the study is explicitly limited to **single-class tasks**.
