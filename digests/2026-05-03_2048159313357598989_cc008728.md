### TL;DR
Matt Pocock argues that “AFK agents” are real but only work when split into a disciplined day-shift/night-shift workflow with heavy planning, automated review, and QA; he says fully unbounded overnight autonomy just produces bad code.

### Key Quote
“AFK agents sometimes produce bad code.”

### Summary
- Matt Pocock says he has been using AFK agents to ship GitHub repos, including:
  - `mattpocock/evalite`
  - `mattpocock/sandcastle`
  - `mattpocock/software-factory` (may have been public later)
- He divides the process into two modes:
  - **Day shift**: planning, review, QA
  - **Night shift**: AFK implementation

- **Day shift workflow**
  - Use `/grill-me` to align with the AI.
  - Use `/to-prd` and `/to-issues` to generate:
    - a PRD as the destination
    - implementation steps as separate tickets that can be done in parallel
  - The PRD is described as requirements gathering, not an actionable step.

- **Night shift workflow**
  - A planner agent checks all tickets and determines what can be worked on and what is blocked.
  - The planner then kicks off multiple sandboxed agents via **Sandcastle** to implement code.
  - An automated reviewer agent checks each set of commits against:
    - the original PRD
    - code quality
  - Commits are pushed to branches and then PR’d to `main`.
  - The planner repeats until all work is completed.

- He emphasizes that **review is crucial** and has saved him many times.
- He says he wants to increase automated review even further, potentially with multiple agents.

- Why AFK agents fail:
  - The original plan was wrong.
  - The plan missed unknown unknowns, forcing bad decisions during implementation.
  - The AI “shat the bed” during implementation or review.
  - The codebase itself is poor, so feedback loops don’t adequately signal success or failure.

- **QA / correction loop**
  - Day shift part 2 is QA:
    - QA all created branches
    - Create follow-up issues
    - Possibly edit the original PRD if the destination changes
  - This can take as long as planning.
  - After QA, the night shift runs again.
  - Important code is still manually reviewed in PRs because he считает the PR UI is still the best available workflow.

- **Wake-up calls / reality checks**
  - If you let AI run all night without planning, it will produce bad code.
  - Most of his loops finish before bedtime; the overnight part is mainly catching up.
  - He uses AFK agents so he can automate review and stop caring about latency.
  - He runs day and night shift in parallel.
  - He says he cannot plan far ahead, so he uses working code to inform planning and aggressively QA whatever lands.

- **Thread context / response**
  - The quoted reply from **Ronan Berder (@hunvreus)** says he’s skeptical of claims that AI builders are “running 20 agents overnight” to build real products.
  - His stance: many of those claims are probably exaggerated; maybe some people build internal tools or disposable software, but building software for actual users is a different story.
