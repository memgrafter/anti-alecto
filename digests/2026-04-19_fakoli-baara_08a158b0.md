### TL;DR
Baara is a Bun-based, Claude Agent SDK-powered personal automation system that lets you create and run scheduled or queued tasks through chat, CLI, and API, with built-in retries, priority queues, and failure triage.

### Key Quote
“An agentic task execution system powered by the Claude Agent SDK.”

### Summary
- **What it is**
  - Baara is an open-source task execution system for personal automation.
  - The name means **“work” in Mandinka**.
  - It is centered on a **chat-first interface** where Claude can create, schedule, and execute tasks by natural language.

- **Core workflow**
  - You type something like: *“Create a task that checks my email every morning at 6am”*.
  - Claude uses exposed tools to create the task, set cron scheduling, and choose tools for execution.
  - The system supports both **immediate execution** and **queued execution**.

- **Key features**
  - **Chat-first interface**
    - Exposes **19 MCP tools** through an in-process server.
    - Supports **SSE streaming** for real-time responses.
    - Includes **slash command autocomplete** with **153+ discovered skills**.
    - Has a **plan mode** for structured planning.
    - Persists sessions across reloads.
    - Shows context like task counts, failures, and queue state.
  - **Task management**
    - 3-step creation wizard: **Basics → Execution → Schedule & Tools**.
    - Per-task tool selection, such as **WebSearch, Bash, Read**, etc.
    - Cron presets or custom expressions with human-readable previews.
    - **Run-once guard** to prevent duplicate execution.
    - Configurable system prompt per instance.
  - **Execution engine**
    - Two modes: **Direct** or **Queued**.
    - Priority queues: **P0 critical** through **P3 background**, FIFO within tier.
    - **At-least-once delivery** with configurable retries and exponential backoff.
    - **Failure triage** marks exhausted retries for human attention.
    - Health monitoring for slow or unresponsive jobs.
    - **JSONL execution logs** that are append-only and `jq`-friendly.
  - **Interfaces**
    - **Web UI**: chat plus collapsible context panel for tasks, jobs, queues, logs.
    - **CLI**: commands like `baara tasks`, `baara jobs`, `baara logs`, `baara status`.
    - **Cron**: scheduled execution via Croner.
    - All three use the same service layer.

- **CLI examples**
  - Task commands:
    - `baara tasks list`
    - `baara tasks create --name <n> --prompt <p> [--cron <expr>]`
    - `baara tasks run <name>`
    - `baara tasks toggle <name>`
  - Job commands:
    - `baara jobs list <task>`
    - `baara jobs triage`
    - `baara jobs retry <job-id>`
  - Logs and discovery:
    - `baara logs`
    - `baara logs --level error`
    - `baara commands list`
    - `baara commands search "email"`
    - `baara status`

- **API**
  - Main endpoints include:
    - `POST /api/chat` for Claude chat via SSE
    - `GET/POST /api/tasks`
    - `POST /api/tasks/:id/run`
    - `POST /api/tasks/:id/submit`
    - `GET /api/tasks/:id/jobs`
    - `GET /api/jobs/triage`
    - `GET /api/queues`
    - `GET /api/logs`
    - `GET /api/projects`
    - `GET /api/commands`
    - `GET /api/health`

- **Architecture and storage**
  - Built around **Chat (Claude AI)**, **Context Panel**, and services that flow through **Dispatcher → Executor → SQLite**.
  - Includes a **Queue Manager** used by CLI and Cron.
  - Stores data under `~/.nexus/`, including:
    - `baara.db` for tasks, jobs, queues, and projects
    - `logs/execution.jsonl`
    - `sessions/`
    - `briefings/`

- **Tech stack**
  - Runtime: **Bun**
  - Web framework: **Hono**
  - Database: **bun:sqlite**
  - Scheduler: **Croner**
  - AI: **Claude Agent SDK**
  - Frontend: **Vanilla JS**
  - Tests: **Playwright**

- **Authentication modes**
  - **Subscription mode** (default): uses Claude Pro/Max/Team through Claude Code CLI, no API key needed.
  - **API key mode**: enable with `BAARA_AUTH_MODE=api_key` and `ANTHROPIC_API_KEY`.

- **Security**
  - Optional API key auth (`BAARA_API_KEY`)
  - Exact-origin CORS allowlist
  - CSP and related security headers
  - Rate limiting: **10 calls/min** on execution endpoints
  - Validation with timeouts, retries, and budgets
  - Graceful shutdown and crash recovery for orphaned jobs
  - Security review docs are in `docs/security/`

- **Development**
  - Start with `bun install`, copy `.env.example` to `.env`, then `bun start`.
  - Typecheck with `bunx tsc --noEmit`.
  - Run tests with `bunx playwright test`.
  - The repo claims **39+ test cases**.
