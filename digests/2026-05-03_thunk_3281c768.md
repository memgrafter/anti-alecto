### TL;DR
Thunk is a **headless, amnesic UNIX job queue / sub-processor for large external LLMs**, splitting an external “Brain” from local “Muscle” workers that execute isolated bash-backed tasks via MCP.

### Key Quote
“**The Brain's context stays pristine. The local model makes mistakes, reads stderr, and fixes them in isolation.**”

### Summary
- **Project idea:** Thunk is designed for cases where a large LLM (70B+) should do planning and architecture, while cheaper local workers handle code execution and debugging.
- **Brain / Muscle split:**
  - **Brain** = external LLM such as opencode, OpenHands, Claude, etc.
  - **Muscle** = local daemon (`thunkd`) that runs cheap 9B workers with isolated bash shells and no memory of prior tasks.
- **Core files:**
  - `thunkd.py` — daemon that polls `.thunk/`, runs workers, and writes results.
  - `mcp_server.py` — stdio MCP server exposing `Dispatch_Thunk` to the Brain.
- **How to run:**
  - Activate virtualenv:
    - Windows: `.venv/Scripts/activate`
    - Linux/macOS: `source .venv/bin/activate`
  - Start daemon: `python thunkd.py`
  - Start MCP server: `python mcp_server.py`
  - Install deps if needed: `pip install litellm pydantic python-dotenv mcp`
- **Configuration variables:**
  - `THUNK_MODEL` default `openai/local`
  - `THUNK_API_BASE` default `http://localhost:8080/v1`
  - `THUNK_SHELL` default `msys2/current/usr/bin/bash.exe`
  - `THUNK_BASH_TIMEOUT` default `120`
  - `THUNK_POLL` default `1.0`
  - `THUNK_DISPATCH_TIMEOUT` default `300`
- **Task model:**
  - Tasks are JSON files in `.thunk/` with fields like:
    - `id`
    - `intent`
    - `target_files`
    - `allowed_tools`
    - `status`
  - Example intent: rename `compute_value` to `compute_doubled` in `fixtures/demo_a.py`.
- **Execution flow:**
  - Daemon claims a pending task.
  - Builds an amnesic prompt.
  - Runs the local model in an agentic loop.
  - Worker uses `Execute_Bash` until done.
  - Worker returns a terse result; task becomes `success`.
- **Hierarchical delegation:**
  - Workers can spawn children with `Create_Thunk`.
  - Parent suspends while child runs.
  - Child result is injected back as the tool response, enabling deep delegation.
- **Guardrails:**
  - **Instant Death:** non-zero bash exit marks task `failed` immediately; stderr is not fed back to the model.
  - **Anti-Yapping:** if a worker has `Execute_Bash` but exits with plain text without calling it, the task fails with `[Harness Error]`.
  - **Crash Recovery:** stale `running` tasks are reset to `pending` when the daemon starts.
- **MCP interface:**
  - `Dispatch_Thunk(intent, target_files, allowed_tools?)`
  - Returns terse worker output or `[FAILED]` / `[TIMEOUT]`.
- **Risks and downsides the project itself calls out:**
  - Destructive bash access can corrupt or delete files.
  - Recursive delegation can create resource exhaustion or self-sabotage.
  - “Instant Death” can lose useful debugging context.
  - The system is vulnerable to shell misuse and local hardware stress.
- **Recommended mitigations:**
  - Only run on a pristine git working tree.
  - Use containers or a restricted VM.
  - Verify every completion via git diff before committing.
