### TL;DR
Recursive Language Models (arXiv:2512.24601) push long-context work beyond the transformer window by putting the prompt in a REPL-style context variable, letting the model use code, `llm_query()`, and a terminating `Final` variable to recursively decompose tasks while keeping costs roughly comparable to base model calls.

### Key Quote
"Recursive Language Models (RLMs) solve the fundamental challenge of scaling long-context processing beyond transformer window limits by treating prompts as environment variables in a REPL."

### Summary
- **What this digest is about**
  - A reference summary of the arXiv paper **2512.24601: Recursive Language Models**.
  - The core idea is to avoid stuffing the entire prompt into the model context window.
  - Instead, the prompt lives outside the model as a **context variable** inside a **REPL** environment.

- **Main architectural idea**
  - The root LLM receives only **metadata** about the prompt:
    - length
    - prefix
    - access methods
  - It then generates **code** to manipulate the external prompt variable.
  - The REPL can call a sub-LLM through **`llm_query()`**.
  - The loop continues until a **`Final`** variable is set.

- **Why it matters**
  - This lets the system process inputs far beyond the model’s native context window.
  - The digest claims inputs can reach **up to two orders of magnitude beyond model context windows**.
  - Intermediate results are stored in variables, which helps bypass single-generation token limits.

- **Reported results**
  - Across **four long-context tasks**, RLMs outperform vanilla models and common scaffolds, with up to **2x performance improvement**.
  - A **fine-tuned 8B RLM** achieves **28.3% average improvement over the base model**.
  - It reportedly approaches **GPT-5 quality on three long-context tasks**.
  - Cost stays in the **same order of magnitude as base-model calls**, despite much longer effective context processing.

- **Mechanism breakdown**
  - **Symbolic handle to prompt**: the prompt is stored externally rather than embedded in attention.
  - **Symbolic recursion for decomposition**: code can iterate over chunks and make sub-LLM calls programmatically.
  - **Variable-based state accumulation**: outputs can be appended over many iterations until `Final` is set.

- **Training / distillation details**
  - Fine-tuning used **1,000 trajectories** from a **teacher RLM**.
  - The setup included **programmatic correction for template mistakes**.
  - The digest notes this as the basis for the reported **28.3% average improvement**.

- **Important limitations and open questions**
  - Experiments were limited to **single-level recursion**; deeper recursion might help but could increase latency and error propagation.
  - Costs and robustness at extreme sizes (the digest mentions **10M+ token range** as a boundary concern) are still uncertain.
  - Generalization to more open-ended, less structured long-context tasks remains unproven.
  - The digest flags uncertainty about whether fine-tuning is necessary versus prompt engineering alone.
  - All experiments used **blocking/sequential** sub-calls, so asynchronous execution remains an open optimization question.

- **Named tasks and examples mentioned**
  - **S-NIAH** at **131K tokens** vs **1M tokens**
  - **OOLONG**
  - **BrowseComp+** with **100, 500, 1000 documents**
  - These are used as sanity checks, ablations, and cost-scaling probes.
