### TL;DR
Peter Oh argues that once an AI-assisted project grows beyond a few scripts, “vibe-coding” breaks down and must be replaced by a more disciplined system with explicit roles, traceable history, and human-controlled workflows.

### Key Quote
“AI doesn’t break when it gets smarter. It breaks when the system around it doesn’t scale.”

### Summary
- **Main premise:** The author started building a personal knowledge base using ChatGPT-generated scripts and integrations, but the project grew into a large codebase that could no longer be managed reliably through ad hoc “vibe-coding.”
- **What the project involved:**
  - Ingesting documents, journals, emails, WhatsApp chats, and Notion notes
  - Reasoning over that data with a Custom GPT
  - Using tools like **Supabase**, **Railway**, **VS Code**, **Cursor**, **Antigravity**, **Claude-code**, and **ChatGPT**
- **Why the old approach failed:**
  - Context switching between tools
  - Repeated explanations across agents
  - Token usage and rate limits
  - No canonical source of truth for plans or decisions
  - Refactors became risky as the codebase grew to:
    - **30,000+ lines of code**
    - **150+ scripts/test/source files**
    - **10+ modules**
    - **25+ tables**
- **New working model:** The author reframed AI coding as a team process rather than a single assistant loop:
  1. **Writer** role writes code
  2. **Reader/Architect** role reviews and critiques code
  3. **Orchestrator** role (the human) owns intent and decisions
- **Tool/role mapping in the workflow:**
  - **Codex** = Writer
  - **Claude-code** = Reader/Architect
  - The author remains the orchestrator
  - Work happens in **iTerm2**, not inside IDEs
- **System architecture / source of truth:**
  - **Jira** holds intent, acceptance criteria, and workflow state
  - **GitHub** holds execution history, branches, PRs, and reviews
  - **AI agents** operate through explicit contracts, permissions, and tool access
  - A **bot identity** represents automation rather than an invisible personal account
- **Why this matters:** The setup creates an “execution ledger” — a durable record of why something exists, how it was built, and who validated it.
- **Observed benefits:**
  - Tickets move through the board as a system, not through ad hoc human prompting
  - PRs are generated by one agent and reviewed by another
  - Decisions are recorded and merge feels less like a leap of faith
  - The author reports **30+ Jira tickets completed** by Claude-code and Codex, including bug fixes
- **Connection to the author’s broader project:**
  - The system is especially relevant because the author is building an **agentic memory layer** with schemas, episodic storage, reference tracking, and TTLs
  - A memory system should be created by a process that is itself memorable, inspectable, and reconstructable
- **Core argument:** AI-assisted development fails less because models lack context and more because the surrounding workflow lacks structure, traceability, and shared context.
- **Tradeoffs / limitations:**
  - Slower than demo-style coding
  - More upfront friction
  - Tooling cannot solve everything; infrastructure and permission issues still matter
  - Adds discipline and constraints rather than removing them
- **Conclusion:** Vibe-coding is still useful for prototyping, but for long-lived systems the future is AI-assisted engineering with contracts, roles, review boundaries, rollback paths, and a human in control.
